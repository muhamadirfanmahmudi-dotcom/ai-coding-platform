# Git HTTP Backend - ⏸️ 搁置

## 状态：搁置（暂停）

## 当前进度

### 已完成
1. ✅ `src/services/git-repo.ts` - Git 仓库管理模块
   - createBareRepo / repoExists / getRepoPath / updateServerInfo
   - handleUploadPack / handleReceivePack（spawn git 子进程）
   - syncToDatabase（将 git log 同步到 SQLite）

2. ✅ `src/routes/git.ts` - Git HTTP 路由
   - GET /:orderId/info/refs（仓库发现， pkt-line 格式）
   - POST /:orderId/git-upload-pack（克隆/拉取）
   - POST /:orderId/git-receive-pack（推送）
   - GET /:orderId/HEAD
   - 认证：从 Bearer token 解码 SID，验证开发者身份

3. ✅ `src/index.ts` - 注册 `/git` 路由 + express.raw 中间件

4. ✅ `src/services/database.ts` - 添加 getCommitByHash 方法

5. ✅ `.gitignore` - 添加 repos/

### 已修复的 Bug
1. pkt-line 头格式：从二进制 UInt16BE 改为 ASCII hex 字符串
2. express.json() 拦截 Git 二进制 POST → 添加 express.raw 中间件
3. req.on('data') 与 express.raw 不兼容 → 改用 req.body

### 未解决的问题
- **clone 仍然失败**：info/refs 返回 200 但 git 客户端报 "not valid: is this a git repository?"
- 可能原因：空仓库时 git-upload-pack 需要返回 capabilities 行；info/refs 响应的 pkt-line 格式可能还有边界问题
- 需要对比真实 Git 服务器（如 Gitea）的响应数据来调试

## 恢复时的建议

### 方案 A：继续调试手写协议
- 用 `git http-backend` 或 Gitea 的 HTTP 实现做对比测试
- 重点检查：空仓库的 upload-pack 响应格式

### 方案 B（推荐）：用成熟库替代
- 安装 `isomorphic-git` 或直接包装系统 `git http-backend`
- 减少协议实现的坑
